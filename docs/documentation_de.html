2. Pioneer CAC Serielles Protokoll

2.1 Befehlsformat

Alle Befehle werden im Format <Player-ID>PS<Befehl>\r gesendet.

Beispiel: 1PS001ZSPL\r — waehlt Player 1, laedt Disc 001 und startet die Wiedergabe.

3.3 Module im Detail

server.js — Einstiegspunkt

Initialisiert alle Komponenten: Datenbank → Serielle Verbindung → PlayerManager → Scanner → Express → WebSocket. Graceful Shutdown ueber SIGINT/SIGTERM.

protocol.js — Pioneer-Protokoll

Enthaelt die komplette Protokolldefinition: Modell-Eigenschaften, Befehls-Konstanten, Response-Parser. Der Parser erkennt automatisch Fehlercodes, Modi, Track-/Disc-Nummern, Zeitcodes, TOC-Info und mehr.

serial.js — Serielle Verbindung

Verwaltet die RS-232C-Verbindung mit automatischer Wiederverbindung (alle 5 Sekunden), separaten Befehlsqueues pro Player (200ms Abstand) und Priority-Queue fuer zeitkritische Befehle. Fire-and-Forget-Architektur: Befehle und Antworten sind komplett entkoppelt.

player.js — Player-Manager

Zentrales Modul mit ~800 Zeilen. Verwaltet den Zustand beider Player, pollt den Wechsler in versetzten Intervallen und implementiert vier Wiedergabe-Modi:

  • Continuous Play: Automatischer Wechsel zum anderen Player bei Disc-Ende
  • Gapless Play: Pre-Cueing des anderen Players fuer nahtlose Uebergaenge
  • Shuffle: CD, beide Player oder gesamter Wechsler
  • Playlists: Server-seitige Engine mit intelligentem Dual-Player-Management und CD-Pre-Loading

scanner.js — CD-Scanner

Scannt Slots: CD laden → Warten → Motor starten → TOC abfragen → Daten speichern → CD zuruecklegen. Echtzeit-Fortschritt per EventEmitter/WebSocket. Pausiert das Polling waehrend des Scans.

musicbrainz.js — MusicBrainz-Integration

Rate-limitierte API-Aufrufe (1,1s Abstand). Suche per Freitext, Kuenstler+Titel, Barcode, Track-Anzahl+Dauer oder Disc-ID. Cover-Art ueber das Cover Art Archive. Automatisches Mapping von MusicBrainz-Daten in das interne Format.

database.js — SQLite-Datenbank

WAL-Modus fuer Performance. UPSERT mit COALESCE (nur uebergebene Felder werden aktualisiert). Kaskadierendes Loeschen fuer Tracks. Standard-Einstellungen beim ersten Start.

routes.js — REST-API

50+ Endpunkte fuer Player-Steuerung, Library-CRUD, Cover-Upload, MusicBrainz-Suche, Scanner, Playlists, Favoriten, Bewertungen, History, Suche, Statistiken, Einstellungen und JSON-Import.

websocket.js — WebSocket-Manager

Broadcast aller Zustandsänderungen an alle verbundenen Clients. Sendet initialen Zustand bei Verbindung.

4. Frontend

4.1 Single Page Application

Das Frontend ist eine SPA ohne Build-Tools oder Frameworks. Drei Dateien: index.html (Struktur), app.js (Logik), i18n.js (Uebersetzungen), app.css (Dark Theme).

4.2 Internationalisierung

Drei Ebenen: HTML-Attribute (data-i18n), JavaScript (t('key')), Server-Nachrichten (strukturierte Objekte). ~200 Uebersetzungsschluessel fuer Deutsch und Englisch.

4.3 Echtzeit-Updates

WebSocket liefert sofortige Updates fuer Player-Status, Scanner-Fortschritt und Playlist-Aenderungen. Die Zeitanzeige wird clientseitig interpoliert fuer fluessige Darstellung.

4.4 Cover-Upload

Clientseitige Verarbeitung: Dateiauswahl/Drag&Drop → Canvas-Resize (max 500x500) → JPEG 85% → Base64-Upload → Server speichert in public/covers/.

5.2 CD-Scan

Browser Server Wechsler | POST /api/scanner/scan | | | { slot: 42 } | | |------------------------>| "1PS042ZS\r" | | |------------------------->| | WS: scanProgress | | | { status: 'loading' } | (wartet auf P01/P02...) | |<========================| | | | "1PSSA\r" | | WS: scanProgress |------------------------->| | { status: 'reading' } | | |<========================| "1PS?Q\r" | | |------------------------->| | | "1PS0112045000\r" | | |<-------------------------| | WS: scanProgress | (TOC: 12 Tracks, 45:00)| | { totalTracks: 12 } | | |<========================| "1PSRJ\r" + "1PSZR\r" | | |------------------------->|

5.3 Playlist mit Dual-Player

Playlist: [Slot 42/Track 3, Slot 101/Track 1, Slot 42/Track 7] Schritt 1: Lade CD 42 in Player 1, spiele Track 3 Gleichzeitig: Lade CD 101 in Player 2 (Pre-Load) Schritt 2: CD 101 bereits in Player 2 Stoppe Player 1, spiele Track 1 auf Player 2 Player 1 hat noch CD 42 (kein Pre-Load noetig) Schritt 3: CD 42 noch in Player 1 Stoppe Player 2, spiele Track 7 auf Player 1

6. Datenbank-Details

6.1 Schema

cds (
  slot INTEGER PRIMARY KEY,     -- 1-500
  disc_id TEXT,                 -- MusicBrainz Disc-ID
  title TEXT, artist TEXT,      -- Album-Metadaten
  year TEXT, genre TEXT,
  total_tracks INTEGER,
  total_duration_seconds INTEGER,
  cover_url TEXT,               -- Pfad zum Cover
  musicbrainz_release_id TEXT,
  barcode TEXT, label TEXT, country TEXT,
  created_at TEXT, updated_at TEXT
)

tracks (
  slot INTEGER REFERENCES cds(slot) ON DELETE CASCADE,
  track_number INTEGER,
  title TEXT, artist TEXT,
  duration_seconds INTEGER, isrc TEXT,
  UNIQUE(slot, track_number)
)

playlists, playlist_items, play_history,
favorites, ratings, settings

6.2 WAL-Modus

Write-Ahead Logging fuer bessere Performance bei gleichzeitigem Lesen und Schreiben.

6.3 UPSERT mit COALESCE

Nur uebergebene Felder werden aktualisiert — bestehende Werte bleiben bei NULL erhalten.

7. JSON-Import

7.1 Unterstuetzte Formate

Array von CDs, Objekt mit cds-Array oder Objekt mit Slot-Nummern als Keys. Automatische Feld-Zuordnung fuer verschiedene Bezeichnungen (slot/cd_number/nr, title/album/name, etc.).

7.2 Vorschau-UI

Editierbare Tabelle mit Status-Badges (Neu/Update/Warnung), aenderbaren Slot-Nummern (Auto-Swap bei Duplikaten), Einzelauswahl per Checkbox und Mojibake-Reparatur.

8. Deployment

8.1 Systemd-Service

[Service]
Type=simple
User=pi
WorkingDirectory=/opt/cac-controller
ExecStart=/usr/bin/node server.js
Restart=always
RestartSec=5
SupplementaryGroups=dialout
Environment=NODE_ENV=production

8.2 Installation

git clone https://github.com/NFDiJee/cac-controller.git /opt/cac-controller
cd /opt/cac-controller
npm install
sudo cp cac-controller.service /etc/systemd/system/
sudo systemctl enable --now cac-controller

9. Sicherheit

  • Keine Authentifizierung (nur fuer LAN gedacht)
  • Kein HTTPS (unverschluesselt)
  • Empfehlung: Firewall, kein Port-Forwarding, bei Bedarf Nginx mit Basic Auth

10. Bekannte Einschraenkungen

  1. Keine Track-Offsets per serieller Schnittstelle: Der Pioneer liefert nur Gesamtdauer und Track-Anzahl. Individuelle Track-Laengen nur ueber MusicBrainz.
  2. Mechanische Geschwindigkeit: CD-Laden dauert 10-30 Sekunden. Vollstaendiger Scan ≈ mehrere Stunden.
  3. Single-Changer: Aktuell wird ein Wechsler gesteuert. Multi-Changer (Hub-Architektur) ist geplant.
  4. Kein Audio-Streaming: Audioausgang ist analog am Geraet. Kein digitales Streaming.
