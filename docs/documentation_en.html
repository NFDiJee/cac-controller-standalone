2. Pioneer CAC Serial Protocol

2.1 Command Format

All commands are sent in the format <Player-ID>PS<Command>\r.

Example: 1PS001ZSPL\r — selects player 1, loads disc 001, and starts playback.

3.3 Module Details

server.js — Entry Point

Initializes all components: Database → Serial Connection → PlayerManager → Scanner → Express → WebSocket. Graceful shutdown via SIGINT/SIGTERM.

protocol.js — Pioneer Protocol

Contains the complete protocol definition: model properties, command constants, response parser. The parser automatically recognizes error codes, modes, track/disc numbers, time codes, TOC info, and more.

serial.js — Serial Connection

Manages the RS-232C connection with automatic reconnection (every 5 seconds), separate command queues per player (200ms delay), and priority queue for time-critical commands. Fire-and-forget architecture: commands and responses are completely decoupled.

player.js — Player Manager

Central module (~800 lines). Manages both player states, polls the changer at staggered intervals, and implements four playback modes:

  • Continuous Play: Automatic switch to the other player on disc end
  • Gapless Play: Pre-cueing of the other player for seamless transitions
  • Shuffle: Current CD, both players, or entire changer
  • Playlists: Server-side engine with intelligent dual-player management and CD pre-loading

scanner.js — CD Scanner

Scans slots: Load CD → Wait → Start motor → Query TOC → Save data → Return CD. Real-time progress via EventEmitter/WebSocket. Pauses polling during scan.

musicbrainz.js — MusicBrainz Integration

Rate-limited API calls (1.1s interval). Search by free text, artist+title, barcode, track count+duration, or disc ID. Cover art via Cover Art Archive. Automatic mapping of MusicBrainz data to internal format.

database.js — SQLite Database

WAL mode for performance. UPSERT with COALESCE (only provided fields are updated). Cascading deletes for tracks. Default settings on first start.

routes.js — REST API

50+ endpoints for player control, library CRUD, cover upload, MusicBrainz search, scanner, playlists, favorites, ratings, history, search, statistics, settings, and JSON import.

websocket.js — WebSocket Manager

Broadcasts all state changes to all connected clients. Sends initial state on connection.

4. Frontend

4.1 Single Page Application

The frontend is a SPA without build tools or frameworks. Four files: index.html (structure), app.js (logic), i18n.js (translations), app.css (dark theme).

4.2 Internationalization

Three levels: HTML attributes (data-i18n), JavaScript (t('key')), server messages (structured objects). ~200 translation keys for German and English.

4.3 Real-time Updates

WebSocket delivers instant updates for player status, scanner progress, and playlist changes. Time display is interpolated client-side for smooth rendering.

4.4 Cover Upload

Client-side processing: File selection/Drag&Drop → Canvas resize (max 500x500) → JPEG 85% → Base64 upload → Server saves to public/covers/.

5. Data Flow

5.1 Player Control

Browser Server Changer | | | | POST /api/player/1/play | | |------------------------>| | | { ok: true } | | |<------------------------| "1PSPL\r" | | |------------------------->| | | | | | "1PSP04\r" | | |<-------------------------| | | | | WS: playerState | (Polling: ?P -> P04) | | { mode: 'P04' } | | |<========================| |

5.2 CD Scanning

Browser Server Changer | POST /api/scanner/scan | | | { slot: 42 } | | |------------------------>| "1PS042ZS\r" | | |------------------------->| | WS: scanProgress | | | { status: 'loading' } | (waiting for P01/P02..)| |<========================| | | | "1PSSA\r" | | WS: scanProgress |------------------------->| | { status: 'reading' } | | |<========================| "1PS?Q\r" | | |------------------------->| | | "1PS0112045000\r" | | |<-------------------------| | WS: scanProgress | (TOC: 12 tracks, 45:00)| | { totalTracks: 12 } | | |<========================| "1PSRJ\r" + "1PSZR\r" | | |------------------------->|

5.3 Playlist with Dual Player

Playlist: [Slot 42/Track 3, Slot 101/Track 1, Slot 42/Track 7] Step 1: Load CD 42 into Player 1, play Track 3 Simultaneously: Load CD 101 into Player 2 (pre-load) Step 2: CD 101 already in Player 2 Stop Player 1, play Track 1 on Player 2 Player 1 still has CD 42 (no pre-load needed) Step 3: CD 42 still in Player 1 Stop Player 2, play Track 7 on Player 1

6. Database Details

6.1 Schema

cds (
  slot INTEGER PRIMARY KEY,     -- 1-500
  disc_id TEXT,                 -- MusicBrainz Disc ID
  title TEXT, artist TEXT,      -- Album metadata
  year TEXT, genre TEXT,
  total_tracks INTEGER,
  total_duration_seconds INTEGER,
  cover_url TEXT,               -- Path to cover
  musicbrainz_release_id TEXT,
  barcode TEXT, label TEXT, country TEXT,
  created_at TEXT, updated_at TEXT

tracks (
  slot INTEGER REFERENCES cds(slot) ON DELETE CASCADE,
  track_number INTEGER,
  title TEXT, artist TEXT,
  duration_seconds INTEGER, isrc TEXT,
  UNIQUE(slot, track_number)
)

playlists, playlist_items, play_history,

6.2 WAL Mode

Write-Ahead Logging for better performance with concurrent reads and writes.

6.3 UPSERT with COALESCE

Only provided fields are updated — existing values are preserved when NULL.

7. JSON Import

7.1 Supported Formats

Array of CDs, object with cds array, or object with slot numbers as keys. Automatic field mapping for various naming conventions (slot/cd_number/nr, title/album/name, etc.).

7.2 Preview UI

Editable table with status badges (New/Update/Warning), editable slot numbers (auto-swap on duplicates), per-CD checkboxes, and mojibake repair.

9. Security

  • No authentication (designed for LAN use only)
  • No HTTPS (unencrypted communication)
  • Recommendation: Firewall, no port forwarding, optionally Nginx with Basic Auth

10. Known Limitations

  1. No track offsets via serial interface: The Pioneer only provides total duration and track count. Individual track lengths only via MusicBrainz.
  2. Mechanical speed: Loading a CD takes 10-30 seconds. Full scan ≈ several hours.
  3. Single changer: Current version controls one changer. Multi-changer (hub architecture) is planned.
  4. No audio streaming: Audio output is analog on the device. No digital streaming.
